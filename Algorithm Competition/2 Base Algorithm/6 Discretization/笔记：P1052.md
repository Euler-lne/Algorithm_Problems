# 为什么你的代码不能AC —— 错误分析与修正笔记

你之前写的代码思路是 **路径压缩 + 单调队列优化 DP**，但由于几个关键错误，导致无法通过全部测试点。下面逐一分析问题，并与正确解法对比。

---

## 1. 石子未排序

### 错误表现
```cpp
for (int i = 0; i < m; i++, pre = cur) {
    scanf("%d", &cur);
    int d = cur;
    if (i != 0) d -= pre;
    // ...
}
```
- 代码直接按输入顺序处理石子距离，但题目 **不保证石子位置是递增的**。
- 如果输入顺序是 `5 2 3`，那么 `d` 的计算就会出错（例如 `5 - 2 = 3`，但实际上石子位置是 `2,3,5`，距离应为 `2,1,2`）。

### 正确做法
读入所有石子后，必须 **排序**：
```cpp
sort(stones, stones + m);
```

## 2. 路径压缩方法不安全

### 错误表现
```cpp
if (d > t) d = d % t + t;
```
- 这个压缩公式来源于某些题解，但 **并不总是正确**。当 `S` 和 `T` 不互质或步长范围特殊时，压缩后的距离可能导致 **原本可达的位置变得不可达**，从而影响 DP 结果。
- 例如：`S=4, T=5`，原始距离 `12` 是可到达的，但 `12 % 5 + 5 = 7`，压缩后 `7` 却不可达（因为步长 4,5 无法组成 7），导致漏算石子。

### 正确做法
采用更安全的压缩阈值：
- 相邻石子间距超过 `T*(T-1)`（最大 90）时，直接压缩为 `90`。
- 最后一段到终点的距离超过 `100` 时，压缩为 `100`（因为青蛙可能跳过终点，多留一点空间）。
```cpp
const int COMPRESS_STONE = 90;
const int COMPRESS_END   = 100;

int d = stones[i] - last;
if (d > COMPRESS_STONE) d = COMPRESS_STONE;
```

## 3. 未处理 S == T 特殊情况

### 错误表现
- 代码没有针对 `S == T` 做任何特殊处理。当步长固定时，青蛙只能落在 `S` 的倍数的位置上，DP 的转移会变得非常特殊，直接用通用 DP 可能得出错误结果（因为压缩后很多位置不可达，但 DP 仍会尝试从不可达状态转移）。

### 正确做法
特判 `S == T`，直接统计这些位置上的石子数：
```cpp
if (S == T) {
    int ans = 0;
    for (int i = 0; i < M; ++i) {
        int pos; scanf("%d", &pos);
        if (pos % S == 0) ans++;
    }
    printf("%d\n", ans);
    return 0;
}
```

## 4. 单调队列使用错误

### 错误表现
```cpp
while (!dq.empty() && f[dq.back()] > f[index - s])
    dq.pop_back();
dq.push_back(index - s);
while (!dq.empty() && dq.front() < index - t)
    dq.pop_front();
value += f[dq.front()];
```
- 当 `index < s` 时，你直接将 `value` 加上 `MAX_VALUE`，但 `MAX_VALUE` 仅 200，可能小于真正的 INF，导致后续转移错误。
- 更重要的是，单调队列中 **允许 INF 值被加入**。当 `f[index-s]` 为 INF 时，它也会被插入队列，而 `dq.front()` 可能取出这个 INF，导致后续 DP 值异常小（因为 INF 加上小值仍为 INF，但逻辑上不应从不可达状态转移）。
- 窗口边界控制也有问题：窗口应是 `[index - T, index - S]`，但你的代码用 `index - s` 和 `index - t` 混用，且未正确排除不可达的状态。

### 正确做法
- 只在 `f[i-S]` **可达**（即不是 INF）时才加入队列。
- 或者，由于 `T-S+1 ≤ 10`，直接使用简单循环取最小值，避免单调队列带来的复杂性和错误。
```cpp
for (int i = 1; i <= newL + T; ++i) {
    f[i] = INF;
    for (int k = S; k <= T; ++k) {
        if (i - k >= 0 && f[i - k] != INF) {
            f[i] = min(f[i], f[i - k] + (hasStone[i] ? 1 : 0));
        }
    }
}
```

## 5. 终点处理不当

### 错误表现
```cpp
int value = j == dist[i] - 1 ? 1 : 0; // 强制将最后一个点视为石子
// ...
printf("%d", f[f.size() - 1] - 1);
```
- 代码将压缩后的 **终点** 也当作石子处理（`value = 1`），最后输出时减 1 来修正。但这样做的假设是：终点一定被标记为石子，而题目保证终点没有石子。
- 如果终点恰好与石子重合（题目保证不会），或者压缩后终点位置恰好有石子（但实际没有），这种修正就会出错。而且这种做法掩盖了真正的问题，不是健壮的解法。

### 正确做法
- 不强行标记终点为石子，而是在 DP 后从 `[newL, newL + T]` 中取最小值（因为青蛙可能跳过终点）。
```cpp
int ans = INF;
for (int i = newL; i <= newL + T; ++i) ans = min(ans, f[i]);
```

## 6. 数组大小与内存访问

### 错误表现
- 原代码使用 `vector<int> f`，但在 `index < s` 时直接 `value += MAX_VALUE`，未考虑可能越界。实际 `f` 大小由压缩后的总点数决定，但 `dist` 的生成方式可能导致总点数远小于预期，从而在单调队列中访问 `f[index - s]` 时可能越界。

### 正确做法
- 确定压缩后总长度后，分配足够大的数组（如 `newL + T + 5`），确保所有索引安全。
- 使用 `vector<int> f(newL + T + 5, INF)` 并明确大小。

## 总结：正确解法要点

1. **排序石子** → 确保距离计算正确。
2. **安全压缩** → 用固定阈值（90 和 100），避免 `d % t + t` 的不确定性。
3. **特判 S==T** → 直接统计，简化处理。
4. **DP 转移** → 使用简单循环（步长范围小），避免单调队列带来的错误。
5. **终点处理** → 取 `[newL, newL+T]` 最小值，不强行标记石子。
6. **数组大小** → 根据压缩后的总长度动态分配，确保安全。

你的代码思路方向正确，但上述细节导致无法 AC。对照标准解法，修正这些错误后，即可通过全部测试点。
